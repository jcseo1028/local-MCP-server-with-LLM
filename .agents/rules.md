# Rules

## 1. Minimal Change Principle

- 요청을 충족하는 **최소한의 변경**만 수행한다.
- 관련 없는 코드를 수정하거나 정리하지 않는다.
- 하나의 변경은 하나의 목적만 가진다.
- 금지 사항:
  - 요청에 포함되지 않은 파일 수정
  - 변경 대상이 아닌 모듈의 코드 수정
  - 기존 동작을 변경하지 않는 포맷팅·네이밍 변경

## 2. No Unnecessary Refactoring

- 동작하는 코드를 스타일, 구조, 성능 개선 목적으로 변경하지 않는다.
- 리팩터링은 사용자가 명시적으로 요청한 경우에만 수행한다.
- "더 나은 방식"이라는 이유만으로 기존 코드를 재작성하지 않는다.

## 3. Contract-First Modification

- 모듈 인터페이스, 데이터 구조, API를 변경할 때 반드시 아래 순서를 따른다:
  1. `.agents/contracts.md`를 먼저 갱신한다.
  2. `.agents/modules.md`의 해당 모듈 의존 관계를 확인·갱신한다.
  3. 이후 구현 코드를 수정한다.
- 계약에 정의되지 않은 인터페이스를 구현에서 사용하지 않는다.

## 4. Scope Adherence

- `.agents/modules.md`에 정의되지 않은 모듈을 생성하지 않는다.
- `.agents/contracts.md`에 정의되지 않은 인터페이스를 추가하지 않는다.
- 추측에 기반한 기능 추가를 금지한다.
- 변경 범위는 요청에 명시된 모듈과 파일로 한정한다.

## 5. Dependency Control

- `modules.md`에 명시된 의존 관계만 허용한다.
- 새 의존 관계가 필요하면 `modules.md`를 먼저 갱신한다.
- 순환 의존을 금지한다.

## 6. Incremental & Testable

- 모든 변경은 증분적이어야 한다 — 한 번에 대규모 변경을 하지 않는다.
- 각 변경은 독립적으로 검증 가능해야 한다.
- 변경 전후 동작 차이를 명확히 설명할 수 있어야 한다.

## 7. Session Finalization

- 작업 세션 종료 시 아래 내용을 `.agents/changes/`에 기록한다:
  - **결정 사항**: 세션 중 내린 설계/구현 결정
  - **변경 요약**: 수정된 파일과 변경 내용
  - **미결 항목**: 다음 세션에서 처리할 사항
- 파일명 형식: `YYYY-MM-DD-<short-description>.md`

## 8. Documentation Sync

- 구현 변경 시 관련 `.agents/` 문서를 동일 세션 내에서 현행화한다.
- 문서와 구현 사이의 불일치를 허용하지 않는다.

## 9. LLM 코드 생성 품질 보증 (v2.6.6)

### 9.1 리팩토링 범위 명확화
- **금지**: public/private 메서드 정의 삭제
- **금지**: 클래스/구조체 필드 삭제 (상수/enum 제외)
- **금지**: public API 시그니처 변경
- **금지**: 클래스 상속/인터페이스 구현 변경
- **허용**: using 정리, 내부 로직 최적화, 변수명 개선, 주석 정리, 코드 스타일 정규화

### 9.2 대용량 파일 처리 (500줄 이상)
- 800줄 이상 파일은 `qwen2.5-coder:32b` 모델 우선 사용
- MaxPerFileChars는 파일 크기에 비례하여 동적 결정 (8KB ~ 20KB)
- 2000줄 이상은 청크 분할 처리 고려

### 9.3 생성 코드 완성도 검증
- **메서드 보존율**: 변경 후 메서드 수 ≥ 변경 전 × 80% (공개 메서드 기준)
- **구문 검증**: C# 컴파일 오류 0개
- **실패 시**: 자동 거부 및 사용자 알림

### 9.4 프롬프트 명확화
- 리팩토링 도구는 금지사항 섹션에 제약 명시
- "using 정리만" 같은 좁은 범위 명시적 기록
- 리팩토링 도구는 단독 실행, 다른 기능과 혼합 금지
